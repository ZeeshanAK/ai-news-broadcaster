## Phase 1 — Feature Identification & Architecture

### 1.1 Feature list (prioritized)

**Core (MVP — build this first):**
- Multi-source news fetching (RSS feeds to start; APIs later)
- LLM-based relevance filtering (the "is this actually news" classifier)
- LLM-based summarization
- Text-to-speech generation
- Telegram delivery
- Scheduling (morning cron)

**High value, build second:**
- Deduplication (same story from 3 sources shouldn't repeat)
- Source credibility weighting
- Structured logging + error handling/retries
- A simple dashboard UI (mock data first, wired up second)

**Nice-to-have (v2, don't block on these):**
- User preference settings (topics to include/exclude)
- Historical archive + search
- Multiple delivery channels (email, Discord)
- Analytics on which stories got read

**Why this ordering matters:** the biggest AI-engineering risk in this project is the filtering/summarization pipeline — that's the part hiring managers will actually probe. Everything else (UI, hosting) is standard software engineering you already know. Spend disproportionate time on the LLM pipeline quality.

### 1.2 Recommended tech stack

| Layer | Choice | Why |
|---|---|---|
| Backend | **Python + FastAPI** | Async-native (good for I/O-bound scraping/API calls), typed, first-class LLM/ML ecosystem, auto-generates OpenAPI docs (nice portfolio touch) |
| Frontend | **React + Vite**, or **FastAPI + HTMX** if you want to stay Python-only | React if you want broader job-market relevance; HTMX if you want to ship faster and keep one language. Given this is portfolio #1, I'd lean **React** — most AI-eng job postings still expect basic frontend fluency and React looks better on GitHub than HTMX to most reviewers |
| Local LLM runtime | **Ollama** | Simplest local serving story, good model catalog, OpenAI-compatible-ish API, plays nicely with your 8GB VRAM if you pick the right model size |
| Local LLM model | **Llama 3.1 8B or Qwen2.5 7B (Q4_K_M quantized)** | Fits comfortably in 8GB VRAM with room for context; good instruction-following for classification/summarization tasks |
| TTS | **Coqui TTS (XTTS-v2)** locally, or **Edge TTS** (free, cloud, no GPU needed) | XTTS-v2 sounds more natural but is heavier — test both against your GPU budget. Edge TTS is a great fallback since it's free and needs zero local compute |
| Scheduling | **APScheduler** (in-process) for local dev → **GitHub Actions cron** or **cron-job.org** for deployed version | Simple, no extra infra needed |
| Database | **SQLite** locally → **Supabase (Postgres)** free tier for deployment | SQLite = zero setup for dev; Supabase gives you a real Postgres instance, auth, and a UI for free, and the migration path SQLite→Postgres via SQLAlchemy is trivial if you use an ORM from day one |
| ORM | **SQLAlchemy 2.0 + Alembic** | Keeps you portable between SQLite and Postgres, migrations tracked in git |

Suggested module layout:
```
app/
  ingestion/
    rss_fetcher.py       # feedparser-based RSS pulls
    api_fetcher.py       # HN API, arXiv API, etc.
  pipeline/
    filter.py            # LLM classification: relevant / not relevant
    summarizer.py        # LLM summarization
    dedup.py             # embedding-based near-duplicate detection
  delivery/

    telegram_bot.py       # python-telegram-bot send logic
  scheduling/
    scheduler.py          # APScheduler jobs
  api/
    routes/               # FastAPI routers: news, settings, logs
  models/                 # SQLAlchemy models
  core/
    config.py             # env-based settings (pydantic-settings)
    llm_client.py          # thin wrapper around Ollama API
```

**API design pattern:** REST, resource-oriented (`/articles`, `/summaries`, `/settings`, `/logs`), with one background-job trigger endpoint (`/pipeline/run`) so you can manually kick the pipeline from the dashboard during dev/demo instead of waiting for the cron.

**LLM integration point:** wrap the inference call behind a single `llm_client.py` interface with two implementations — `OllamaClient` (local dev only) and `CloudLLMClient` (what actually runs in production, see 1.5 below) — selected by an env var. This is what makes Phase 4's "must run without local GPU" requirement trivial: swap the client, not the pipeline code.

### 1.5 Important distinction: dev-time code generation vs. runtime inference

These are two separate concerns and shouldn't be conflated:

- **Ollama / Aider / Claude Code** — used only *while you're developing*, on your laptop, to generate the application code itself. They never run as part of the deployed app.
- **The deployed app's own inference and TTS calls** — once live and running on a schedule with no GPU behind it, every filtering/summarization/TTS call needs to hit a free cloud API instead.

**Free LLM inference for the deployed pipeline:**

| Provider | Free tier (no card required) | Role in this project |
|---|---|---|
| **Groq** | ~14,400 requests/day, OpenAI-API-compatible, serves Llama 3.1 8B / 3.3 70B / 4 Scout, Qwen3 32B | **Primary** — fast, generous, more than covers one daily batch of articles |
| **Google Gemini API** | 1,500 requests/day on Gemini Flash, up to 1M-token context, no expiry | **Fallback** — also useful if you ever want to reason over a very large batch of articles in a single call |
| **OpenRouter** | Aggregates many `:free`-suffixed models behind one key | **Secondary fallback** — insurance against a single provider changing its free catalog |

Free tiers shift their limits and model catalogs without much warning, so the fallback chain matters more than picking "the best" provider — build `CloudLLMClient` to try Groq first and fall back to Gemini on a 429/error, rather than hardcoding a single provider.

**Free TTS for the deployed pipeline:**

| Provider | Free tier | Role in this project |
|---|---|---|
| **Edge TTS** | Free, uncapped, no signup (uses Microsoft Edge's online voices via an open-source wrapper) | **Primary** — pragmatic default for a short daily broadcast |
| **ElevenLabs** | ~10,000 characters/month (~10 min of audio) | **Quality upgrade** once the pipeline works end-to-end — a few hundred words/day comfortably fits this cap |
| **Google Cloud TTS** | ~1M characters/month, neural voices | Alternative if you want higher volume/quality and don't mind a Google Cloud account |

**Telegram delivery:** no free-tier concern here — the Telegram Bot API is free with no meaningful rate limit for one message/day. `python-telegram-bot` + a bot token from BotFather is all you need.

**Recommended default stack for the deployed app:**

| Stage | Primary | Fallback |
|---|---|---|
| Filtering | Groq (Llama 3.1 8B or 3.3 70B) | Gemini Flash |
| Summarization | Groq | Gemini Flash (for large-batch/long-context cases) |
| TTS | Edge TTS | ElevenLabs free tier |
| Delivery | Telegram Bot API | — |

This maps directly onto the module layout in 1.3: `llm_client.py` exposes `GroqClient` and `GeminiClient` behind a common interface, selected by config with automatic fallback on error; `tts.py` does the same for `EdgeTTSClient` and `ElevenLabsClient`.

### 1.4 Database schema (initial draft)

```sql
-- sources: what you're pulling from
CREATE TABLE sources (
    id INTEGER PRIMARY KEY,
    name TEXT NOT NULL,
    url TEXT NOT NULL,
    type TEXT NOT NULL,          -- 'rss' | 'api'
    active BOOLEAN DEFAULT TRUE,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- raw articles pulled before filtering
CREATE TABLE articles (
    id INTEGER PRIMARY KEY,
    source_id INTEGER REFERENCES sources(id),
    title TEXT NOT NULL,
    url TEXT NOT NULL UNIQUE,
    raw_content TEXT,
    published_at TIMESTAMP,
    fetched_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    category TEXT,                -- 'benchmark' | 'product_launch' | 'low_spec_model' | 'free_tool' | null
    relevance_score REAL,         -- LLM confidence this is "real" news
    is_relevant BOOLEAN
);

-- deduped clusters of the same underlying story
CREATE TABLE story_clusters (
    id INTEGER PRIMARY KEY,
    representative_article_id INTEGER REFERENCES articles(id),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE article_clusters (
    article_id INTEGER REFERENCES articles(id),
    cluster_id INTEGER REFERENCES story_clusters(id),
    PRIMARY KEY (article_id, cluster_id)
);

-- daily digest: the actual output artifact
CREATE TABLE digests (
    id INTEGER PRIMARY KEY,
    digest_date DATE NOT NULL,
    summary_text TEXT NOT NULL,
    audio_path TEXT,               -- local path or object storage URL
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE digest_articles (
    digest_id INTEGER REFERENCES digests(id),
    cluster_id INTEGER REFERENCES story_clusters(id),
    PRIMARY KEY (digest_id, cluster_id)
);

-- delivery logs
CREATE TABLE delivery_logs (
    id INTEGER PRIMARY KEY,
    digest_id INTEGER REFERENCES digests(id),
    channel TEXT NOT NULL,          -- 'telegram'
    status TEXT NOT NULL,           -- 'sent' | 'failed'
    error_message TEXT,
    delivered_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- user preferences (single-user for now, but structured for multi-user later)
CREATE TABLE user_settings (
    id INTEGER PRIMARY KEY,
    telegram_bot_token TEXT,
    telegram_chat_id TEXT,
    schedule_time TEXT DEFAULT '07:00',
    included_categories TEXT       -- JSON array, e.g. '["benchmark","product_launch"]'
);
```

**Design rationale:** separating `articles` (raw ingestion) from `story_clusters` (deduped stories) from `digests` (the daily output) means each pipeline stage is independently testable and re-runnable — important because LLM outputs are non-deterministic, and you'll want to re-run just the summarization stage without re-fetching everything.

### 2.2 Should you use multiple AI agents?

For a project this size, I'd recommend **against** a complex multi-agent setup (e.g., separate agents for code/test/docs coordinating with each other) — the coordination overhead isn't worth it at this scale, and it obscures what you're actually learning. Instead:

- Use **one Aider or Claude Code session per module** (task 1–15 above), each as its own focused conversation/commit. This gives you agent-like separation of concerns without the orchestration complexity.
- Optionally, run a **second, separate session as a reviewer**: after a module is generated, open a fresh Aider/Claude Code session with only "review this diff for bugs and edge cases" as its job. This "generator + reviewer" split is the one multi-agent pattern that reliably pays off at small scale, because a fresh context catches issues the generating session is blind to.

### 2.3 Concrete setup guide (Aider + Ollama)

```bash
# 1. Install Ollama and pull your model
# (download from ollama.com, then:)
ollama pull qwen2.5:7b-instruct-q4_K_M

# 2. Verify it fits and runs well
ollama run qwen2.5:7b-instruct-q4_K_M "Summarize: AI news test."

# 3. Install Aider
pip install aider-chat

# 4. Point Aider at your local Ollama model
export OLLAMA_API_BASE=http://127.0.0.1:11434
aider --model ollama/qwen2.5:7b-instruct-q4_K_M

# 5. Inside Aider, work module-by-module, e.g.:
#    /add app/models/*.py
#    "Generate SQLAlchemy models for the schema in ai-news-broadcaster-plan.md section 1.4"
```

If Aider + a 7–8B local model struggles with multi-file reasoning (common — small local models are noticeably weaker at large-context code tasks than frontier models), that's a legitimate and common trade-off to make explicitly: you can use **Claude Code** for the harder scaffolding tasks (initial project structure, complex modules) and fall back to **Ollama** for the repetitive, well-specified tasks (CRUD endpoints, simple wrappers) — and document that trade-off in your README. That's a mature engineering decision, not a cop-out, and it's worth stating explicitly in an interview.

### 2.4 Alternate dev environment: GitHub Codespaces

Since the deployed pipeline runs entirely on free cloud APIs (Groq/Gemini, Edge TTS/ElevenLabs, Telegram — see 1.5), a GPU-less cloud dev environment like Codespaces works fine for this project whenever you don't have your GPU laptop available.

**One adjustment when working from Codespaces:** Ollama needs a GPU (or a strong CPU) to be usable, and Codespaces machines have neither. Swap to a cloud-based coding tool instead:
- **Claude Code**, or
- **Aider pointed at Groq/OpenRouter** instead of Ollama

Both run fine from the Codespaces browser terminal with no local compute required — you're just extending the same cloud-first approach from 1.5 to your dev tooling.

**Free tier limits to plan around:**
- Personal GitHub accounts get **120 core-hours + 15 GB storage per month**, free.
- This is core-hours, not wall-clock hours — a 2-core machine burns 2 hours of quota per 1 real hour used (~60 hours/month of actual coding time); a 4-core machine burns twice as fast.
- The quota resets monthly. **Explicitly stop** (not just leave idle/suspended) your Codespace when you're done for the day.
- The 15 GB storage cap is separate from compute and persists even when stopped — keep generated audio files and `node_modules`/venv bloat out of git, and delete old Codespaces you're not using.

**Setup so switching machines is seamless:**
1. Commit and push before switching machines (already part of the Phase 3 workflow below).
2. Store real secrets (Groq/Gemini/Telegram API keys) as **Codespaces secrets** in your GitHub repo or account settings — not a local-only `.env` file — so they're available automatically in any new Codespace.
3. Add a `.devcontainer/devcontainer.json` to the repo to pre-configure the Python/Node environment, so a fresh Codespace boots ready to run instead of needing a manual dependency install each time.

## Phase 3 — Implementation Order (guidance, not code)

Since you want the local LLM tooling to generate the actual code, here's the order and what to hand each session:

1. **UI first, with mock data** — give Aider/Claude Code the four screens (dashboard of news cards, filter/sort, summary view, settings page) and a hardcoded JSON fixture matching the `articles`/`digests` schema above. This lets you validate the UX before any backend exists.
2. **Then backend, module by module**, in the task order from 2.1 — each module gets its own session/commit.
3. **After each task**, commit:
   ```bash
   git add .
   git commit -m "feat: <task description>"
   git push origin main
   ```

I'd suggest coming back with the actual generated code from each module as you go (or pasting in tricky Aider/Ollama output) — I can review it, catch issues, or help you debug the LLM prompt design for filtering/summarization, which is usually the trickiest part to get right.
